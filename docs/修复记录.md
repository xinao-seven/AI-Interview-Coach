# AI Interview Coach — 历次 Bug 修复与优化记录

> 记录项目从最初版本到现在经历的每一次关键修复和架构优化。

## 修复 1：虚拟列表布局 Bug — 顶部空白

**症状**：面试回看页面顶部全是空白，只能看到最下面几条消息。

**根因**：VirtualChatList 的 DOM 结构中，phantom 和 viewport 是文档流中的兄弟元素。phantom 的 `height: totalHeight`（如 10000px）将 viewport 推到了 y=10000px 的位置。viewport 内的 item 用 `position: absolute; top: 0` 定位，实际位于容器的 10000px 处，而不是顶部。

```
错误结构：
  <div class="container" style="overflow:auto">
    <div class="phantom" style="height:10000px" />      ← 撑高
    <div class="viewport" style="position:relative">     ← 排在 phantom 后面，起点在 y=10000
      <div style="top:0px"> 消息 </div>                  ← top:0 相对 viewport，实际 y=10000
    </div>
  </div>
```

**修复**：改为单层 inner 结构，用 `minHeight` 撑开滚动条的同时作为绝对定位的参考容器。

```
正确结构：
  <div class="container" style="overflow:auto">
    <div class="inner" style="position:relative; min-height:10000px">
      <div style="position:absolute; top:0"> 消息1 </div>   ← 真正的 y=0
      <div style="position:absolute; top:200"> 消息2 </div>
    </div>
  </div>
```

## 修复 2：重复历史条 + 旧条不可点击

**症状**：完成面试后历史页面出现两条记录，一条正常可回看，一条点击无效。

**根因**：`saveInterviewHistory()` 做了双重保存——先调 `historyStore.addHistory()` 存完整对象，然后又手动操作同一个 localStorage key 存简化对象（无 `session` 字段）。两种格式混在一起，简化对象 `getSession()` 返回 `null`，导致不可点击。

**修复**：删除手动 localStorage 操作，统一走 `historyStore.addHistory()`。

```diff
- // 同时更新旧版兼容数据
- localStorage.getItem('interview-history')
- localStorage.setItem('interview-history', ...)
```

**症状**：不知道在哪里切换 Mock 和真实后端。

**修复**：

- 创建 `.env` 文件，加 `VITE_AI_MODE=mock`（默认 Mock）
- 切换为 `VITE_AI_MODE=real` 即走真实后端
- 后端 `.env.example` 新增 `AI_ENABLE_THINKING` 配置

## 修复 4：DeepSeek 思考过程不显示

**症状**：DeepSeek 模型有思考能力，但评分时看不到任何推理过程。Mock 数据中也没有。

**根因**：后端 `_base_params()` 对 DeepSeek 强制 `extra_body: { thinking: { type: "disabled" } }`，完全扼杀了思考输出。

**修复**：

1. 新增 `AI_ENABLE_THINKING` 配置项，只有设为 `false` 时才禁用思考
2. 新增 `chat_completion_with_thinking()` 函数，自动提取 `reasoning_content` 和 `<thinking>` 标签内容
3. `chat_completion_json()` 返回的 dict 附带 `_thinking` 字段
4. 评分 prompt 新增"先输出 `<thinking>` 块"的指引
5. 前端 `AnswerEvaluation` 新增 `thinking` 字段，`formatEvaluation()` 拼接 `<thinking>` 块
6. MarkdownRenderer 新增 `preprocessThinking()` + `buildThinkingFold()`，用 `<details>` 原生元素折叠展示
7. Mock 数据新增 `thinking` 字段

## 修复 5：Thinking 折叠面板不工作 — HTML 被转义

**症状**：`<div class="thinking-fold" data-folded="true">` 等 HTML 以原始文本形式展示在页面上，折叠功能完全失效。

**根因**：两个问题叠加——

1. `markdown-it` 初始化时 `html: false`，把所有 HTML 标签转义为 `&lt;div&gt;`
2. 折叠面板用了 `onclick` 内联事件，即使 `html: true` 也会被 DOMPurify 剥离

**修复**：

1. `markdown-it` 改为 `html: true`（DOMPurify 兜底安全）
2. 用原生 `<details>` + `<summary>` 替代 `div + onclick` hack
3. DOMPurify 白名单新增 `<details>`、`<summary>`，移除 `onclick`
4. CSS 用 `[open]` 选择器驱动三角图标 `▸`/`▾` 自动切换

---

## 修复 6：SSE 流式评分 — 从假流式到真流式

**症状**：不管是 Mock 还是 Real 模式，评分都是先等 AI 全文返回（3~15 秒），然后才用 `simulateStream` 模拟打字机播放。

**根因**：`handleSubmit()` 中 `await evaluateAnswerApi()` 是阻塞调用，等完整 JSON 返回后才往下走。后端 `/interview/evaluate` 用的也是 `chat_completion_json()`（非流式）。

**修复**：

1. 后端新增 `POST /api/interview/evaluate/stream` SSE 流式端点
2. 前端新增 `evaluateAnswerStreamApi()`，用 `createStreamRequest` 边收边回调
3. `handleSubmit()` 按 Mock/Real 分叉：Mock 保持 `await` + `simulateStream`，Real 走 `evaluateAnswerStreamApi`
4. 抽取 `finishStreaming()` 统一收尾逻辑

---

## 修复 7：流式时原始 JSON 暴露

**症状**：Real 模式 SSE 流式时，AI 输出的原始 JSON（`{"totalScore":85,...}`）直接显示在页面上，等流结束才突然跳变为格式化表格。

**修复**：新增 `_buildStreamingDisplay()` 函数，检测到 `</thinking>` 标签闭合后，将后续 JSON 替换为 `⏳ 正在生成评分...` 占位符。

---

## 修复 8：评分逐字显示 + 布局重构

**需求**：评分部分也要逐字显示，同时从三栏布局改为 AI 对话两栏布局。

**修复**：

1. 修改评分 prompt：AI 输出从 JSON 改为 Markdown 自然语言（`<thinking>` + 优缺点 + 建议 + 追问），不再输出任何评分分数
2. 简化后端流式端点：纯文本推送，仅提取追问
3. Mock 数据去分数化
4. 前端删除右侧评分面板 → 两栏布局
5. `formatEvaluation()` 去分数表格，改为纯文本格式
6. 单题不打分，所有分数统一在最终报告中生成

## 修复 9：消息 ID 碰撞导致追问回答丢失

**症状**：回答追问后，AI 的回复显示为用户气泡（角色错误），且用户的追问回答内容丢失了。

**根因**：`Date.now()` 返回毫秒级整数。Real 模式下，创建用户消息和评分空壳消息之间没有异步等待（`evaluateAnswerStreamApi` 同步返回），两个 `Date.now()` 调用可能在 1ms 内返回相同值，导致消息 ID 碰撞。`updateMessage(id, content)` 通过 `find()` 匹配到第一条同 ID 消息（用户消息），用户消息的 `content` 被评分文本覆盖，角色仍为 `user`。

**修复**：新增全局自增计数器 `_msgSeq`，每条消息 ID 改为 `msg-{timestamp}-{seq}`：

```ts
let _msgSeq = 0;
function _nextMsgId(): string {
    return `msg-${Date.now()}-${++_msgSeq}`;
}
```

替换了全部 6 处 `msg-${Date.now()}` 调用。

## 关键经验教训

| 教训                                               | 具体体现                                                     |
| -------------------------------------------------- | ------------------------------------------------------------ |
| **`Date.now()` 不能保证唯一性**                    | 同一毫秒内多次调用产生碰撞 → 用自增计数器兜底                |
| **`position: absolute` 的参考容器位置至关重要**    | phantom 在文档流中把 viewport 推到下方 → 单层 inner 结构解决 |
| **不要同时维护两套持久化逻辑**                     | historyStore + 手动 localStorage → 统一走 Store              |
| **`html: false` 会转义所有内联 HTML**              | 折叠面板完全失效 → `html: true` + DOMPurify 兜底             |
| **`onclick` 会被 DOMPurify 剥离**                  | ≡用原生 `<details>` + CSS `[open]` 替代 JS 交互              |
| **非流式 API + simulateStream ≠ 真正的边收边渲染** | 用户干等 15 秒 → 改用 SSE 流式端点                           |
| **AI 输出 JSON 与流式展示天然矛盾**                | JSON 流式丑陋 → 改为 Markdown 自然语言评价                   |
