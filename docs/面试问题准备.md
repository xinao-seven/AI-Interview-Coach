> 基于简历中描述的三个核心亮点，逐层拆解面试官可能追问的问题。

## 一、SSE 流式对话渲染模块

### Q1：Fetch + ReadableStream 实现 SSE 的具体流程是什么？为什么不直接用 EventSource？

**回答要点**：

- **EventSource 的局限**：只支持 GET 请求，无法传 JSON Body、无法自定义请求头（如 Authorization），不支持 POST 方式提交对话上下文
- **Fetch + ReadableStream 流程**：
    1. `fetch(url, { method: 'POST', body: JSON.stringify(payload), signal })` 发起请求
    2. `response.body.getReader()` 获取 `ReadableStreamDefaultReader`
    3. 循环 `reader.read()` 逐块读取 `Uint8Array`
    4. `TextDecoder.decode(chunk, { stream: true })` 流式解码（`stream: true` 防止多字节字符截断）
    5. 缓冲区 `buffer` 按 `\n` 分割行，最后一行保留等待补全（解决 TCP 分片半包）
    6. 匹配 `data: ` 前缀提取 SSE 事件，`data: [DONE]` 表示流结束
    7. `reader.releaseLock()` 释放锁，`AbortController.abort()` 取消请求

### Q2：TCP 分片导致的半包/沾包问题具体怎么解决？

**回答要点**：

- **半包**：一条 SSE 消息 `data: {"content":"你好"}\n\n` 被切到两个 TCP chunk 中
- **沾包**：多个 SSE 消息在一个 chunk 中
- **解法**：维护 `buffer` 字符串，每收到 chunk 追加到 buffer → `buffer.split('\n')` 分割 → **最后一行 `pop()` 后放回 buffer**（可能不完整）→ 只处理确认完整的行
- **多字节字符**：`TextDecoder.decode(value, { stream: true })` 内部维护解码状态，UTF-8 3 字节中文字符被截断时暂存，等下一 chunk 补全后再输出

### Q3：AbortController 的中断控制是怎么实现的？组件卸载时怎么处理？

**回答要点**：

- `new AbortController()` 创建控制器，`signal` 传入 `fetch()`
- 调用 `controller.abort()` → `fetch` 的 Promise reject `AbortError`
- catch 中判断 `err.name !== 'AbortError'` 才报错（用户主动取消不报错）
- 组件 `onUnmounted` 中调用 `abort()` 防止内存泄漏
- `finally` 块中 `reader.releaseLock()` 确保锁释放（无论成功/失败/取消）

### Q4：流式输出如何实现"滚动跟随"？用户浏览历史时如何暂停自动滚动？

**回答要点**：

- **滚动跟随**：每次 `onMessage` 回调中调用 `scrollToBottom()` → `container.scrollTo({ top: container.scrollHeight })`
- **浏览历史暂停**：计算 `distanceFromBottom = scrollHeight - scrollTop - clientHeight`，> 50px 认定用户在浏览历史 → 暂停自动滚动
- **回到底部**：显示浮动"回到底部"按钮，点击恢复 `scrollToBottom('smooth')`
- 新消息到来时，只有在"粘底状态"才自动滚到底部

### Q5：流式输出中如果网络中断或超时，怎么处理？

**回答要点**：

- `fetch` 内置超时配置（默认 60s），超时触发 `AbortController.abort()`
- 支持重试机制：`createStreamSession` 封装了 `maxRetries` + `retryDelay`，网络抖动自动重连
- 重试次数耗尽后调用 `onError` 回调，上层展示错误提示
- Mock 模式下用 `simulateStream`（`setTimeout` 递归 + 随机延迟）模拟真实流式效果

## 二、动态高度虚拟列表

### Q1：虚拟列表的核心原理是什么？为什么长列表需要它？

**回答要点**：

- **问题**：1000 条 DOM 节点 → Layout/Paint 耗时随数量线性增长，滚动帧率下降，内存占用高
- **原理**：只渲染可视区域 ± overscan 范围内的 DOM 节点，其余用占位高度替代
- **三步走**：
    1. **计算总高度**：每项高度累加 → `totalHeight` 放入幽灵元素 `minHeight` 撑开滚动条
    2. **二分查找可视区**：根据 `scrollTop` 二分查找 `startIndex`，`scrollTop + containerHeight` 二分查找 `endIndex`
    3. **绝对定位渲染**：可见项用 `position: absolute; top: offset` 定位

### Q2：动态高度怎么测量？ResizeObserver 的工作流程？

**回答要点**：

- 每个渲染项挂载时，`ResizeObserver.observe(el)` 监听其高度变化
- 回调中读取 `entry.contentRect.height`，写入 `heightCache[id]`
- `heightCache` 是 `ref<Record<string, number>>({})`，变化触发 `offsets` 和 `visibleItems` 重新计算
- 未测量项用 `estimatedItemHeight`（默认 120px）作为预估值
- **关键细节**：`heightCache.value = { ...heightCache.value, [id]: h }` 创建新对象触发 Vue 响应式

### Q3：动态高度下，新测量的高度导致项位置偏移怎么处理？

**回答要点**：

- 这是虚拟列表的经典"抖动问题"
- `offsets` 是 computed，依赖 `heightCache`，任何高度变化自动触发重新计算
- `visibleItems` 的 `top` 值随之更新，项自动移动到正确位置
- 为避免频繁抖动：ResizeObserver 批量处理（同一帧内多个 entry 遍历完后再更新 `heightCache`）
- 实际场景中，消息高度通常在流式完成后才稳定，ResizeObserver 只在内容变化时触发

### Q4：为什么用 minHeight 而不是 height 来撑开滚动条？

**回答要点**：

- 这是你实际踩过的坑
- **错误做法**：phantom 和 viewport 作为兄弟元素 → phantom `height: totalHeight` → viewport 排在phantom 后面 → viewport 的 y 起点 = totalHeight → item 的 `top: 0` 实际在 `totalHeight` 位置 → 顶部空白
- **正确做法**：统一 inner 包裹层，`minHeight: totalHeight` 撑开滚动条，item 直接绝对定位在 inner 内 → `top: offsets[i]` 就是真实的 y 坐标

### Q5：虚拟列表怎么处理内容还在流式增长的消息？

**回答要点**：

- ResizeObserver 持续监听，消息内容增长 → `contentRect.height` 变化 → `heightCache` 更新 → `offsets` 重算 → 下面所有项自动下移
- 配合 `watch(() => props.items, ... , { deep: true, flush: 'post' })`，流式增长实时反映
- 如果在"粘底"模式，`nextTick` 后自动 `scrollToBottom()`

## 三、Markdown 安全渲染与思考链折叠

### Q1：Markdown 渲染的完整管道是什么？安全怎么保证？

**回答要点**：

- **管道**：原始文本 → 思考预处理（`<thinking>` → `<details>`） → `markdown-it.render()` → `DOMPurify.sanitize()` → `v-html` 输出
- **安全防线**：
    1. `markdown-it` 转义 Markdown 语法为 HTML
    2. `DOMPurify.sanitize()` 白名单模式：只允许 `ALLOWED_TAGS` 中的标签，剥离 `<script>`、`<iframe>`、`onerror` 等
    3. 代码块复制按钮中的数据用 `base64` 编码存储，防止 XSS
- 降级策略：渲染异常时回退为纯文本转义（`&lt;` `&gt;` `&amp;`）

### Q2：思考链折叠是怎么实现的？为什么用 `<details>` 而不是 JS 控制？

**回答要点**：

- **预处理阶段**：正则匹配 `<thinking>...</thinking>` 或 `【思考】...【/思考】` → 替换为 `<details class="thinking-fold"><summary>🧠 思考过程...</summary>...</details>`
- **为什么用 `<details>`**：
    - 原生 HTML5 元素，浏览器内置折叠/展开逻辑，零 JS
    - `open` 属性自动追踪展开状态，CSS 用 `[open]` 选择器切换图标
    - 不会像 `onclick` 那样被 DOMPurify 剥离
- **摘要截取**：取思考内容前 80 字符作为 `summary` 预览文本

### Q3：代码高亮是怎么实现的？复制按钮怎么防 XSS？

**回答要点**：

- `markdown-it` 自定义 `fence` 渲染器，使用 `highlight.js` 的 `hljs.highlight(code, { language })` 做语法高亮
- 主题使用 `atom-one-dark.css`（暗色背景，适合技术对话）
- **复制按钮防 XSS**：代码内容 `base64` 编码存入 `data-code` 属性 → 点击时 `atob` 解码 → `navigator.clipboard.writeText()`
- 不用 `data-code="原始代码"` 是防止代码中包含 `"` 或 `&quot;` 破坏 HTML 结构

### Q4：DeepSeek 等推理模型的 thinking 是怎么提取和展示的？

**回答要点**：

- **后端**：`chat_completion_with_thinking()` 函数同时获取 `reasoning_content`（API 原生字段）和 `<thinking>` 标签内容，两者合并
- **传递**：`chat_completion_json()` 返回的 dict 附带 `_thinking` 字段 → 路由层提取放入 response 的 `thinking` 字段
- **前端**：`evaluateAnswerApi()` 从 response 中提取 `thinking` → 存入 `AnswerEvaluation.thinking` → `formatEvaluation()` 拼接 `<thinking>` 块
- **展示**：MarkdownRenderer 预处理将 `<thinking>` 转为 `<details>` 折叠面板，用户可点击展开查看 AI 完整推理过程

## 四、综合架构问题

### Q1：项目的整体架构是怎样的？前后端怎么通信？

**回答要点**：

```
Vue3 SPA (Vite, Port 5173)
  ├── Pinia Stores (resume / interview / report / mistake / history)
  ├── API Layer (Mock/Real 双模式，VITE_AI_MODE 切换)
  ├── SSE Utils (fetch + ReadableStream 自研封装)
  └── Components (VirtualChatList / MarkdownRenderer / AbilityRadarChart / ScoreCard)
         │
         │ Vite Proxy: /api → http://127.0.0.1:5000
         ▼
Flask (Port 5000)
  ├── /api/resume/parse      — AI 简历解析
  ├── /api/interview/questions — AI 题目生成
  ├── /api/interview/evaluate  — AI 回答评分（含 thinking）
  ├── /api/interview/report    — AI 报告生成
  ├── /api/project/polish      — 项目描述优化
  ├── /api/project/questions   — 项目深挖题
  └── /api/chat/stream         — SSE 流式对话
         │
         ▼
OpenAI 兼容 API (GPT-4o / DeepSeek)
```

### Q2：Pinia Store 怎么设计的？为什么不用 Vuex？

**回答要点**：

- 按领域拆分 5 个独立 Store：`resume`、`interview`、`report`、`mistake`、`history`
- `interview` Store 最复杂：管理面试会话全生命周期（config → questions → messages → evaluations → followUpStates）
- **选择 Pinia 的原因**：完全 TypeScript 类型推导、去掉 Mutation 概念简化 API、Composition API 风格与 Vue3 一致、无需 `modules` 嵌套
- 每个 Store 独立管理 `localStorage` 持久化（`saveToStorage` / `loadFromStorage`），面试中途刷新不丢进度

### Q3：追问机制的状态机是怎么设计的？

**回答要点**：

- **数据结构**：`followUpStates: Record<questionId, { question: string; mainAnswer: string }>`
- **状态判断**：
    - `hasActiveFollowUp` = `currentQuestion.id in followUpStates` → 决定 UI 显示追问提示
    - `hasCurrentAnswer` = `evaluations 中有该题评分 AND !hasActiveFollowUp` → 决定显示输入框还是"下一题"按钮
- **生命周期**：主问题回答 → 评分 → AI 返回 `followUpQuestion` → `setFollowUp()` → 追问回答 → 追问评分 → `clearFollowUp()` → 下一题
- **追问评分上下文**：传递 `{ isFollowUp: true, followUpQuestion, mainAnswer }` 让 AI 结合原始答案评估追问质量

### Q4：Mock/Real 双模式是怎么实现的？

**回答要点**：

- **切换**：`VITE_AI_MODE` 环境变量 → `const isMock = import.meta.env.VITE_AI_MODE !== 'real'`
- **API 层分发**：`parseResumeApi()` 内部 `if (isMock) return mockParseResume(data)` 否则发 HTTP 请求
- **接口签名统一**：Mock 和 Real 的返回值类型完全一致，调用方（页面组件）无感知
- **Mock 真实感**：随机延迟 500-900ms、关键词匹配简历解析、按面试模式分题库、思考过程模拟

### Q5：这个项目你从中学到了什么？如果再迭代，你会怎么改进？

**回答要点**：

- **学到的**：TCP 流式协议细节（半包/沾包/多字节编码）、虚拟列表的工程落地细节（布局结构的重要性）、AI 应用中的防御性编程（容错解析、降级策略）
- **改进方向**：
    1. 消息存储从 `localStorage` 迁移到 `IndexedDB`（更大容量 + 索引查询）
    2. 虚拟列表支持向上预加载历史消息（当前只支持向下滚动）
    3. 引入 `Web Worker` 处理大量消息的 Markdown 预渲染
    4. 添加 PWA 支持实现离线使用
    5. 后端增加对话记忆（RAG），让 AI 记住之前轮次的对话上下文
